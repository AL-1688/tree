# 需求文档

## 引言

本需求文档针对 GitHub 客户端桌面应用的「文件夹上传功能完善」模块。在现有基础上，增强上传进度可视化、断点续传支持、并发上传控制、上传队列管理等功能，并实现大文件智能上传策略（小于 50MB 分块上传，大于 50MB 提示使用 Git LFS）。

## 词汇表

- **应用程序**: 指 GitHub 客户端管理工具桌面应用
- **上传任务**: 用户发起的一次文件夹上传操作，包含多个文件上传
- **上传队列**: 待上传文件的有序集合，支持并发控制和优先级管理
- **断点续传**: 上传中断后能够从上次中断的位置继续上传，而非重新开始
- **并发上传**: 同时进行多个文件的上传操作，提高整体上传效率
- **分块上传**: 使用 GitHub Git Data API 将大文件分块处理并上传
- **Git LFS**: Git Large File Storage，用于管理大文件的 Git 扩展
- **上传状态快照**: 记录上传任务当前状态的持久化数据，用于恢复上传

## 需求

### 需求1: 上传进度可视化

**用户故事:** AS 用户,我想要实时看到文件夹上传的详细进度,以便了解上传状态和预估完成时间

#### Acceptance Criteria

1. WHEN 上传任务开始,应用程序 SHALL 显示上传进度面板,包含总体进度条、当前文件名和已上传文件数
2. THE 进度面板 SHALL 显示上传速度（KB/s 或 MB/s）和预估剩余时间
3. THE 进度面板 SHALL 显示每个文件的上传状态（等待中、上传中、已完成、失败）
4. WHEN 用户点击进度面板,应用程序 SHALL 展开显示完整文件列表及其状态
5. THE 进度面板 SHALL 支持最小化和展开两种显示模式
6. IF 上传过程中出现错误,应用程序 SHALL 在对应文件旁显示错误图标和简要错误信息

### 需求2: 断点续传支持

**用户故事:** AS 用户,我想要在网络中断或应用意外关闭后能够继续之前的上传,以便避免重复上传已完成的部分

#### Acceptance Criteria

1. THE 应用程序 SHALL 在上传过程中定期保存上传状态快照到本地存储
2. WHEN 上传任务被中断（网络断开、应用崩溃、手动暂停）,应用程序 SHALL 保留已上传文件的状态记录
3. WHEN 应用程序重新启动,应用程序 SHALL 自动检测并提示恢复未完成的上传任务
4. WHEN 用户选择恢复上传任务,应用程序 SHALL 跳过已成功上传的文件,仅上传未完成部分
5. THE 应用程序 SHALL 保存每个文件的 SHA 值用于验证文件是否已正确上传
6. IF 远程仓库文件与本地文件内容一致（SHA 匹配）,应用程序 SHALL 自动跳过该文件
7. WHEN 断点续传恢复成功,应用程序 SHALL 显示本次恢复上传的文件数量和跳过的文件数量

### 需求3: 并发上传控制

**用户故事:** AS 用户,我想要控制上传的并发数量,以便在保证上传效率的同时不影响其他网络活动

#### Acceptance Criteria

1. THE 应用程序 SHALL 支持并发上传,默认并发数为 3
2. THE 设置页面 SHALL 提供并发数配置选项,范围 1-10
3. WHEN 用户修改并发数,应用程序 SHALL 动态调整当前上传任务的并发数
4. THE 应用程序 SHALL 根据网络响应时间自动调整并发数（自适应模式）
5. IF 连续 3 次请求超时或失败,应用程序 SHALL 自动降低并发数
6. WHEN 网络状况恢复良好,应用程序 SHALL 逐步恢复并发数至设定值
7. THE 上传进度面板 SHALL 显示当前活跃的上传连接数

### 需求4: 上传队列管理

**用户故事:** AS 用户,我想要管理多个上传任务的执行顺序,以便合理安排上传资源

#### Acceptance Criteria

1. THE 应用程序 SHALL 支持创建多个上传任务并加入队列
2. THE 上传管理页面 SHALL 显示所有任务（进行中、等待中、已完成、已取消）
3. WHEN 用户创建新上传任务,应用程序 SHALL 根据当前状态决定立即执行或加入等待队列
4. THE 用户 SHALL 能够暂停、恢复、取消队列中的任务
5. THE 用户 SHALL 能够调整等待队列中任务的优先级顺序
6. WHEN 当前任务完成或取消,应用程序 SHALL 自动开始队列中的下一个任务
7. THE 应用程序 SHALL 限制同时进行的上传任务数不超过 3 个

### 需求5: 暂停和恢复上传

**用户故事:** AS 用户,我想要暂停正在进行的上传任务,以便在需要时释放网络带宽或稍后继续

#### Acceptance Criteria

1. THE 上传进度面板 SHALL 提供「暂停」按钮
2. WHEN 用户点击暂停按钮,应用程序 SHALL 停止提交新的上传请求并等待当前正在进行的上传完成
3. WHEN 上传任务暂停,应用程序 SHALL 保存当前状态快照
4. THE 上传进度面板 SHALL 提供「继续」按钮用于恢复暂停的任务
5. WHEN 用户点击继续按钮,应用程序 SHALL 从上次暂停的位置恢复上传
6. IF 应用程序在暂停状态下关闭,应用程序 SHALL 保留任务状态以便下次恢复

### 需求6: 大文件智能上传策略

**用户故事:** AS 用户,我想要应用程序自动为大文件选择合适的上传方式,以便确保上传成功并避免超时

#### Acceptance Criteria

1. THE 应用程序 SHALL 判断文件大小,对小于 50MB 的文件使用分块上传
2. WHEN 文件大小超过 50MB,应用程序 SHALL 显示 Git LFS 建议提示
3. THE Git LFS 提示对话框 SHALL 包含「了解 Git LFS」链接和「继续直接上传」选项
4. IF 用户选择「继续直接上传」超过 50MB 的文件,应用程序 SHALL 使用分块上传方式
5. THE 分块上传 SHALL 支持进度回调,实时报告上传进度
6. IF 分块上传过程中发生错误,应用程序 SHALL 自动重试最多 3 次
7. THE 应用程序 SHALL 在设置页面提供大文件阈值配置选项（默认 50MB）

### 需求7: 文件冲突处理

**用户故事:** AS 用户,我想要在上传时处理目标仓库中已存在的同名文件,以便避免意外覆盖重要内容

#### Acceptance Criteria

1. WHEN 开始上传前,应用程序 SHALL 检测目标仓库中是否存在同名文件
2. IF 检测到同名文件,应用程序 SHALL 显示冲突处理对话框
3. THE 冲突处理对话框 SHALL 提供「覆盖」、「跳过」、「保留两者（重命名）」选项
4. THE 用户 SHALL 能够选择「应用到所有冲突」以批量处理
5. WHEN 用户选择「重命名」,应用程序 SHALL 自动为上传文件添加序号后缀
6. THE 应用程序 SHALL 记住用户在当前上传任务中的冲突处理偏好

### 需求8: 上传错误处理与重试

**用户故事:** AS 用户,我想要在上传失败时获得清晰的错误信息并能重试,以便成功完成上传

#### Acceptance Criteria

1. WHEN 单个文件上传失败,应用程序 SHALL 在文件列表中标记失败状态并显示错误原因
2. THE 失败文件列表 SHALL 提供「重试」按钮用于单独重试失败的文件
3. WHEN 上传任务完成后存在失败文件,应用程序 SHALL 显示「重试所有失败文件」按钮
4. IF 错误是网络超时,应用程序 SHALL 自动重试最多 3 次,每次间隔递增
5. IF 错误是权限不足,应用程序 SHALL 显示具体的权限要求并引导用户检查 Token 权限
6. IF 错误是 GitHub API 速率限制,应用程序 SHALL 显示剩余等待时间并自动等待重试
7. THE 应用程序 SHALL 记录错误日志到本地文件,包含时间戳、文件名、错误类型和详情

### 需求9: 上传任务历史

**用户故事:** AS 用户,我想要查看历史上传任务的记录,以便了解过去的上传活动

#### Acceptance Criteria

1. THE 应用程序 SHALL 保存上传任务的历史记录,包含任务时间、目标仓库、文件数量、状态
2. THE 上传管理页面 SHALL 显示最近 30 天的上传历史
3. THE 用户 SHALL 能够搜索和筛选历史记录
4. WHEN 用户点击历史记录条目,应用程序 SHALL 显示该任务的详细信息
5. THE 应用程序 SHALL 保存历史记录 30 天后自动清理
